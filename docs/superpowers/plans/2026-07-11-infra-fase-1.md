# Infra de producción — Fase 1 — Plan de implementación

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Separar el despliegue en un servidor público (nginx con SSL y balanceador) y uno privado (API x2, panel, BD), y hacer que la API deje de correr como superusuario de Postgres.

**Architecture:** Dos `docker-compose` (público y privado). nginx termina TLS, es el único dueño de la cabecera HSTS, y balancea entre dos réplicas reales de la API mediante un `upstream`. La configuración de nginx usa plantillas con `envsubst`, de modo que el mismo fichero sirve en local (contenedores) y en AWS (IP privada de la otra EC2). En la BD se separan dos roles: uno admin para las migraciones de Alembic y uno limitado para la API en runtime.

**Tech Stack:** nginx 1.27 (imagen oficial, plantillas `envsubst`), OpenSSL (certificado autofirmado), PostgreSQL 16 (scripts de `docker-entrypoint-initdb.d`), uvicorn (`ProxyHeadersMiddleware`), Flask (`ProxyFix` de Werkzeug), Alembic.

## Global Constraints

- **No se toca el entorno local.** `docker-compose.yml` (desarrollo) sigue funcionando igual, con `postgres` como usuario de BD. Los cambios de menor privilegio solo aplican al stack de producción.
- **Los 192 tests deben seguir en verde** al terminar cada tarea que toque código de `api/`.
- **Nunca registrar secretos.** Se mantiene la regla del grupo D: ni contraseñas, ni hashes, ni tokens en los logs.
- **Rutas exactas:** todo lo nuevo de infraestructura vive en `infra/`.
- `docker-compose.prod.yml` queda **obsoleto** y se elimina en la Tarea 6; su contenido se reparte entre el compose público y el privado.
- Comandos de test: desde `api/`, con `.venv/bin/python -m pytest`.

### Task 1: La API respeta la IP real del cliente detrás del proxy

Al poner nginx delante, `request.client.host` pasa a ser la IP de **nginx**, no la del usuario. Eso rompe dos cosas ya construidas:

- **Rate limiting (grupo B):** `get_remote_address` devolvería siempre la misma IP, así que todos los usuarios compartirían un único cupo de 20/min. El anti-fuerza-bruta se volvería una autodenegación de servicio.
- **Auditoría (grupo D):** cada línea registraría la IP de nginx. Inútil para investigar un incidente.

La solución es leer `X-Forwarded-For`, pero **solo cuando la API está realmente detrás de un proxy**. Si se confiara siempre en esa cabecera, cualquiera podría falsificarla para saltarse el rate limiting.

**Files:**
- Modify: `api/app/main.py`
- Test: `api/tests/test_proxy_headers.py` (crear)

**Interfaces:**
- Consumes: `app.audit.ip_de(request)` (del grupo D), `app.rate_limit.limiter` (del grupo B).
- Produces: variable de entorno `TRUSTED_PROXIES`. Vacía o ausente = no se confía en la cabecera (comportamiento por defecto, seguro). Con valor = lista de proxies de confianza que pasa a `ProxyHeadersMiddleware` (`"*"` en el compose privado, donde solo nginx alcanza la API).

- [ ] **Step 1: Escribir el test que falla**

```python
"""
Detrás de nginx, la IP del cliente llega en X-Forwarded-For. Sin esto, el rate
limiting y la auditoría verían siempre la IP del proxy.

Pero confiar en esa cabecera sin estar detrás de un proxy permitiría a cualquiera
falsificar su IP para saltarse el rate limiting: por eso solo se confía cuando
TRUSTED_PROXIES está definida.
"""
import importlib
import logging

import pytest
from fastapi.testclient import TestClient

from app import audit


def _app_con(monkeypatch, trusted_proxies):
    """Reconstruye la app con el valor dado de TRUSTED_PROXIES."""
    if trusted_proxies is None:
        monkeypatch.delenv("TRUSTED_PROXIES", raising=False)
    else:
        monkeypatch.setenv("TRUSTED_PROXIES", trusted_proxies)

    import app.main
    importlib.reload(app.main)
    return app.main.app


def test_confia_en_x_forwarded_for_cuando_hay_proxy(monkeypatch, caplog):
    caplog.set_level(logging.INFO, logger="auditoria")
    aplicacion = _app_con(monkeypatch, "*")

    with TestClient(aplicacion) as cliente:
        cliente.post(
            "/auth/login",
            json={"correo": "nadie@demo.com", "password": "mala"},
            headers={"X-Forwarded-For": "203.0.113.7"},
        )

    # La auditoría registra la IP real del cliente, no la del proxy.
    assert "ip=203.0.113.7" in caplog.text


def test_ignora_x_forwarded_for_sin_proxy(monkeypatch, caplog):
    """Sin proxy la cabecera es falsificable: debe ignorarse."""
    caplog.set_level(logging.INFO, logger="auditoria")
    aplicacion = _app_con(monkeypatch, None)

    with TestClient(aplicacion) as cliente:
        cliente.post(
            "/auth/login",
            json={"correo": "nadie@demo.com", "password": "mala"},
            headers={"X-Forwarded-For": "203.0.113.7"},
        )

    assert "ip=203.0.113.7" not in caplog.text
```

### Task 2: El panel Flask construye URLs https detrás del proxy

nginx termina el TLS y habla con Flask por HTTP plano. Flask cree entonces que el esquema es `http` y construye sus redirecciones con `http://`. El login redirige tras autenticar, así que el usuario saldría de HTTPS a HTTP en el primer clic.

`ProxyFix` hace que Flask lea `X-Forwarded-Proto` y `X-Forwarded-For`.

**Files:**
- Modify: `web/app/app.py`

**Interfaces:**
- Consumes: cabeceras `X-Forwarded-Proto` / `X-Forwarded-For` que nginx envía (Tarea 4).


```python
# nginx termina el TLS y habla con Flask por HTTP plano. Sin esto, Flask creería
# que el esquema es http y construiría las redirecciones (p. ej. la del login)
# con http://, sacando al usuario de HTTPS en el primer clic.
```

### Task 4: Postgres crea el usuario limitado de la API

Hoy la API se conecta como `postgres`, **superusuario**: un fallo de inyección o una credencial filtrada daría control total de la base. El usuario limitado solo puede leer y escribir filas, nunca alterar el esquema.

La trampa: si se otorgan permisos sobre las tablas existentes, las que Alembic cree **después** no quedarán accesibles. `ALTER DEFAULT PRIVILEGES` resuelve eso de raíz.

**Files:**
- Create: `infra/db/init/01-usuario-app.sh`

**Interfaces:**
- Consumes: variables `DB_APP_USER` / `DB_APP_PASSWORD` del entorno del contenedor de Postgres.
- Produces: el rol limitado en la BD. La API se conecta con `DB_USER=$DB_APP_USER` (Tarea 6).

- [ ] **Step 1: Crear el script de init**

Crear `infra/db/init/01-usuario-app.sh`:

```bash
#!/bin/bash
# Crea el usuario con el que la API se conecta en runtime.
#
# Postgres ejecuta este script UNA sola vez, al inicializar un volumen nuevo.
# Por eso la base de producción se crea desde cero y el entorno local no se toca.
#
# El usuario NO recibe CREATE sobre el esquema: no puede crear ni alterar tablas.
# Eso lo hacen las migraciones, que corren con el usuario admin (POSTGRES_USER).
set -e

psql -v ON_ERROR_STOP=1 --username "$POSTGRES_USER" --dbname "$POSTGRES_DB" <<-EOSQL
    CREATE ROLE "${DB_APP_USER}" LOGIN PASSWORD '${DB_APP_PASSWORD}';

    GRANT CONNECT ON DATABASE "${POSTGRES_DB}" TO "${DB_APP_USER}";
    GRANT USAGE ON SCHEMA public TO "${DB_APP_USER}";

    -- La clave del montaje: las tablas que Alembic cree DESPUÉS heredan estos
    -- permisos automáticamente. Sin esto habría que re-otorgar permisos tras
    -- cada migración, y alguien lo olvidaría.
    ALTER DEFAULT PRIVILEGES FOR ROLE "${POSTGRES_USER}" IN SCHEMA public
        GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO "${DB_APP_USER}";

    ALTER DEFAULT PRIVILEGES FOR ROLE "${POSTGRES_USER}" IN SCHEMA public
        GRANT USAGE, SELECT ON SEQUENCES TO "${DB_APP_USER}";
EOSQL

echo "Usuario limitado '${DB_APP_USER}' creado (sin permisos de DDL)."
```

- [ ] **Step 1: Crear el script del certificado**

Crear `infra/nginx/generar-certificado.sh`:

```bash
#!/bin/bash
# Genera un certificado autofirmado para poder servir HTTPS sin dominio.
#
# El cifrado es real; lo que el navegador no puede verificar es la identidad, así
# que mostrará una advertencia. Es lo esperado hasta que haya dominio.
#
# Cuando exista el dominio, certbot escribirá fullchain.pem y privkey.pem en esta
# misma ruta y nginx no necesitará ningún cambio (ver la guía de despliegue).
set -e

DESTINO="$(dirname "$0")/certs"
mkdir -p "$DESTINO"

if [ -f "$DESTINO/fullchain.pem" ]; then
    echo "Ya existe un certificado en $DESTINO. No se toca."
    exit 0
fi

openssl req -x509 -nodes -newkey rsa:2048 -days 365 \
    -keyout "$DESTINO/privkey.pem" \
    -out "$DESTINO/fullchain.pem" \
    -subj "/C=MX/ST=Hidalgo/L=Pachuca/O=Sistema de Torneos/CN=${1:-localhost}"

chmod 600 "$DESTINO/privkey.pem"
echo "Certificado autofirmado generado en $DESTINO (CN=${1:-localhost})."
```

- [ ] **Step 3: Crear la plantilla de nginx**

Crear `infra/nginx/templates/default.conf.template`. La imagen oficial de nginx procesa los ficheros de `/etc/nginx/templates/*.template` con `envsubst` al arrancar, sustituyendo las variables de entorno. Así el mismo fichero sirve en local y en AWS.

```nginx
# Las direcciones de los backends llegan por entorno:
#   En local: API_1=api1:8000       (nombres de contenedor)
#   En AWS:   API_1=10.0.2.10:8001  (IP privada de la EC2 privada)
# Así este fichero no se edita a mano al desplegar, que es donde se cuelan errores.

upstream api_backend {
    server ${API_1};
    server ${API_2};
}

upstream panel_backend {
    server ${WEB_1};
}

# Todo el tráfico en claro se redirige a HTTPS.
server {
    listen 80;
    server_name _;

    # Reservado para el desafío HTTP-01 de certbot cuando exista el dominio.
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

server {
    listen 443 ssl;
    http2 on;
    server_name _;

    ssl_certificate     /etc/nginx/certs/fullchain.pem;
    ssl_certificate_key /etc/nginx/certs/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    # nginx, que es quien termina el TLS, es el ÚNICO dueño de HSTS.
    # La API y el panel también la emiten (grupo D) y, si llegaran las dos, el
    # navegador descartaría la política entera: el requisito parecería cumplido
    # sin estarlo. Por eso se descarta la de la app antes de emitir la nuestra.
    proxy_hide_header Strict-Transport-Security;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # Cabeceras que necesitan los backends:
    #   X-Forwarded-For   -> la IP real del cliente (rate limiting y auditoría)
    #   X-Forwarded-Proto -> para que Flask construya redirecciones https
    #
    # CRÍTICO: se SOBRESCRIBE con $remote_addr, no se anexa.
    # Con $proxy_add_x_forwarded_for (que anexa), un atacante que enviara
    # "X-Forwarded-For: 9.9.9.9" haría que nginx reenviara "9.9.9.9, <ip_real>",
    # y uvicorn (trusted_hosts="*") toma el PRIMER valor: el falsificado.
    # Rotándolo en cada petición, el atacante anularía por completo el rate
    # limiting y envenenaría la auditoría. Verificado explotable.
    # nginx es el único proxy del sistema, así que no hay cadena que preservar.
    proxy_set_header Host              $host;
    proxy_set_header X-Real-IP         $remote_addr;
    proxy_set_header X-Forwarded-For   $remote_addr;
    proxy_set_header X-Forwarded-Proto $scheme;

    # La API, repartida entre las dos réplicas.
    # La barra final de proxy_pass quita el prefijo /api antes de reenviar.
    location /api/ {
        proxy_pass http://api_backend/;
    }

    # El panel de administración.
    location / {
        proxy_pass http://panel_backend;
    }
}
```

```yaml
      TRUSTED_PROXIES: "*"
    env_file: .env
    depends_on:
      db:
        condition: service_healthy
    networks:
      - privada

  api2:
    <<: *api
    container_name: torneos_api2

  web:
    build: ./web
    container_name: torneos_web
    restart: unless-stopped
    command: gunicorn app.app:app --bind 0.0.0.0:5000 --workers 4
    environment:
      FLASK_ENV: production
      TRUSTED_PROXIES: "*"
      # El panel habla con la API por la red interna, sin pasar por nginx.
      API_URL: http://api1:8000
    env_file: .env
    depends_on:
      - api1
    networks:
      - privada

volumes:
  pgdata:

networks:
  # En AWS esta red es la red privada de la VPC. En local, una red Docker
  # externa que la simula (ver la guía de verificación).
  privada:
    external: true
    name: torneos_privada
```

Nota sobre los puertos: en local, nginx alcanza `api1:8000` y `api2:8000` por nombre de contenedor dentro de la red `torneos_privada`. En AWS, la EC2 privada publica esos puertos en su interfaz privada (`8001`, `8002`) y nginx apunta a su IP; eso se explica en la guía de despliegue (Tarea 8).

```bash
#!/bin/bash
# Verifica la infraestructura levantada en local (los dos compose).
# -k porque el certificado es autofirmado: eso es esperado sin dominio.
set -u

echo "=== 1. HTTP redirige a HTTPS ==="
curl -s -o /dev/null -w "http://localhost -> %{http_code} (esperado 301)\n" http://localhost/

echo "=== 2. HTTPS responde y el panel carga ==="
curl -sk -o /dev/null -w "https://localhost/login -> %{http_code} (esperado 200)\n" https://localhost/login

echo "=== 3. La API responde a través de nginx ==="
curl -sk https://localhost/api/health; echo

echo "=== 4. EXACTAMENTE una cabecera HSTS (dos la anularían) ==="
total=$(curl -sk -D- -o /dev/null https://localhost/login | grep -ci "^strict-transport-security" || true)
echo "cabeceras HSTS = $total (esperado 1)"

echo "=== 5. El balanceador reparte entre las dos réplicas ==="
for i in $(seq 1 6); do
  curl -sk -o /dev/null https://localhost/api/health
done
echo "peticiones servidas por api1: $(docker logs torneos_api1 2>&1 | grep -c 'GET /health')"
echo "peticiones servidas por api2: $(docker logs torneos_api2 2>&1 | grep -c 'GET /health')"
echo "(ambas deben ser > 0)"

echo "=== 6. Tolerancia a fallos: se apaga una réplica ==="
docker stop torneos_api2 >/dev/null
curl -sk -o /dev/null -w "con api2 caída: %{http_code} (esperado 200)\n" https://localhost/api/health
docker start torneos_api2 >/dev/null

echo "=== 7. La auditoría registra la IP REAL, no la de nginx ==="
curl -sk -o /dev/null -X POST https://localhost/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"correo":"intruso@demo.com","password":"mala"}'
docker logs torneos_api1 2>&1 | grep AUDIT | tail -1
docker logs torneos_api2 2>&1 | grep AUDIT | tail -1

echo "=== 7b. Un atacante NO puede falsificar su IP (regresión de seguridad) ==="
# Si nginx anexara la cabecera en vez de sobrescribirla, uvicorn tomaría este
# valor forjado como IP del cliente: el atacante lo rotaría en cada petición y
# anularía el rate limiting por completo. Verificado explotable en su día.
curl -sk -o /dev/null -X POST https://localhost/api/auth/login \
  -H "Content-Type: application/json" \
  -H "X-Forwarded-For: 9.9.9.9" \
  -d '{"correo":"falsificador@demo.com","password":"mala"}'
if docker logs torneos_api1 2>&1 | grep AUDIT | grep -q "9.9.9.9" || \
   docker logs torneos_api2 2>&1 | grep AUDIT | grep -q "9.9.9.9"; then
  echo "FALLO: la IP falsificada (9.9.9.9) llegó a la auditoría."
  echo "       nginx debe SOBRESCRIBIR X-Forwarded-For con \$remote_addr, no anexar."
else
  echo "correcto: la IP falsificada fue descartada"
fi

echo "=== 8. Postgres NO es accesible desde el host ==="
(exec 3<>/dev/tcp/localhost/5432) 2>/dev/null && echo "ACCESIBLE (mal)" || echo "no accesible (correcto)"

echo "=== 9. La API corre con el usuario LIMITADO, no como superusuario ==="
docker compose --env-file .env.produccion.local -f docker-compose.privado.yml exec -T db \
  psql -U "${DB_ADMIN_USER:-torneos_admin}" -d "${DB_NAME:-torneos}" -tc \
  "SELECT rolname, rolsuper FROM pg_roles WHERE rolname IN ('torneos_app','torneos_admin');"

echo "=== 10. El usuario limitado NO puede hacer DDL ==="
docker compose --env-file .env.produccion.local -f docker-compose.privado.yml exec -T db \
  psql -U torneos_app -d "${DB_NAME:-torneos}" -c "CREATE TABLE intento_ddl (id int);" 2>&1 | head -2
echo "(debe decir 'permission denied' o similar)"
```

````markdown
# Despliegue en AWS

Arquitectura de dos servidores, como exigen los Requerimientos Mínimos del PI:
uno de acceso público y otro privado.

```
Internet ──:443──> EC2 PÚBLICA (nginx: SSL + balanceador)
                        │ red privada de la VPC
                        ▼
                   EC2 PRIVADA (API x2 + panel + PostgreSQL)
                   sin IP pública: inalcanzable desde internet
```

## 1. Red (VPC)

| Recurso | Valor |
|---|---|
| VPC | `10.0.0.0/16` |
| Subred pública | `10.0.1.0/24` — con Internet Gateway |
| Subred privada | `10.0.2.0/24` — sin Internet Gateway |
| NAT Gateway | En la subred pública. Deja que la EC2 privada **salga** a internet (para `docker pull`), pero impide que la alcancen desde fuera. |

## 2. Instancias

| | EC2 pública | EC2 privada |
|---|---|---|
| Subred | pública | privada |
| IP pública | sí (Elastic IP) | **no** |
| Tipo sugerido | t3.small | t3.medium |
| Software | Docker + Docker Compose | Docker + Docker Compose |

## 3. Security Groups (el firewall de AWS)

**`sg-publica`** — entrada:

| Puerto | Origen | Motivo |
|---|---|---|
| 443 | `0.0.0.0/0` | HTTPS |
| 80 | `0.0.0.0/0` | Redirección a HTTPS y desafío de certbot |
| 22 | **solo tu IP** | SSH de administración |

**`sg-privada`** — entrada:

| Puerto | Origen | Motivo |
|---|---|---|
| 8001, 8002 | `sg-publica` | Las réplicas de la API, solo desde nginx |
| 5000 | `sg-publica` | El panel, solo desde nginx |
| 22 | `sg-publica` | SSH saltando desde la pública (bastión) |

**Nunca** se abre el 5432: Postgres solo se alcanza dentro de la propia EC2 privada.

> La Fase 2 añade `ufw` dentro de cada VM como segunda capa, más `fail2ban` para
> monitorear el firewall.

## 6. Certificado real (cuando haya dominio)

Mientras no hay dominio, el certificado es autofirmado: **cifra de verdad**, pero
el navegador avisa de que no puede verificar la identidad. Con un dominio
apuntando a la IP pública, se sustituye por uno real y gratuito de Let's Encrypt:

```bash
docker run --rm \
  -v ./infra/nginx/certs:/etc/letsencrypt/live/tudominio.com \
  -v certbot_www:/var/www/certbot \
  certbot/certbot certonly --webroot -w /var/www/certbot \
  -d tudominio.com --agree-tos -m tu@correo.com

docker compose -f docker-compose.publico.yml restart nginx
```

Certbot escribe `fullchain.pem` y `privkey.pem` en la misma ruta que ya usa
nginx, así que **no hay que tocar la configuración**. Renovación automática:

```bash
0 3 * * * cd /home/ec2-user/sistema-futbol && docker run --rm -v ./infra/nginx/certs:/etc/letsencrypt certbot/certbot renew --quiet && docker compose -f docker-compose.publico.yml restart nginx
```
````
